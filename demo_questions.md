# Demo Questions

These questions are designed to run against the database generated by `scripts/seed_demo.py`.
Each one exercises a specific capability of the system.

---

## Multi-table JOIN + aggregation
**"Who are the top 10 customers by total amount spent?"**

What it shows: the system detects `orders → users` and `order_items → orders` relationships,
auto-expands the table selection, injects the join hints, and the LLM writes a correct
three-table JOIN without being told which tables to use.

Expected SQL shape:
```sql
SELECT u.name, SUM(oi.quantity * oi.unit_price) AS total_spent
FROM users u
JOIN orders o ON o.user_id = u.id
JOIN order_items oi ON oi.order_id = o.id
WHERE o.status = 'completed'
GROUP BY u.id, u.name
ORDER BY total_spent DESC
LIMIT 10
```

## Time-series aggregation → line chart
**"Show me monthly revenue for 2023"**

What it shows: date-based grouping triggers the chart recommender to pick `line` type.
The pipeline trace will show `orders` and `order_items` both selected via relationship expansion.

## Category breakdown → bar chart
**"What is the total revenue by product category?"**

What it shows: three-table join across `order_items → products` + `order_items → orders`
(to filter completed orders). Chart recommender picks `bar`. Good screenshot for GitHub.

## Filtering on a status enum
**"How many orders were returned last year?"**

What it shows: the system correctly uses the `status` column on `orders` and scopes
to a date range — two constraints in one query with no JOIN needed.

---

## Funnel / cohort question
**"Which cities have the most users?"**

What it shows: clean single-table aggregation on `users`. Simple but validates
that the table selector doesn't over-expand when no JOIN is needed.

---

## Product performance
**"What are the 5 best-selling products by units sold?"**

What it shows: `order_items → products` join, SUM on quantity, correctly limited to 5.
Good bar chart output.

## Average order value
**"What is the average order value for completed orders?"**

What it shows: subquery or direct aggregation pattern. Tests whether the LLM
handles `SUM / COUNT` correctly on a filtered dataset.

---

## Schema validator in action (hallucination catch)
If you deliberately ask something that would tempt the LLM to invent a column:

**"Show me total revenue per customer email domain"**

What it shows: forces the LLM to use `SUBSTR` or `INSTR` on `users.email` — a real
column. If it invents a fake column like `users.domain`, the schema validator
catches it before execution and returns a 422 with a suggestion.

---

## Cross-category comparison
**"Compare the average price of Electronics vs Books"**

What it shows: single-table query on `products` with a `CASE` or `GROUP BY category`
filter. Clean, readable SQL, good for demos.

---

## Relationship expansion visible in trace
**"List all orders placed by users from New York"**

What it shows: the question only mentions `orders` and `New York` (a `users.city` value).
The system expands from `orders` → `users` via the relationship graph. The pipeline
trace in the response explicitly shows `relationships_used: ["orders.user_id FK→ users.id"]`.
This is the clearest demo of the schema-aware routing.

## Quick start

```bash
# 1. Generate the database
python scripts/seed_demo.py

# 2. Start the API
DATABASE_URL=sqlite:///dev.db uvicorn app.api.main:app --reload

# 3. Open the UI
streamlit run ui/app.py

# 4. Paste any question above into the text box
```
